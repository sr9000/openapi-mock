## Important findings / issues

1. Per-operation label is `operation="unknown"` in metrics and traces for the tested Echo endpoints.
   - The strict middleware intended to inject the OpenAPI operation name into context and logs exists and an `OperationContext` middleware was added and wired into strict handlers.
   - However, after regeneration / runtime, the operation label remains `unknown`. This indicates the operation name is not resolved earlier enough for requests or that the strict middleware isn't being executed for these paths in the current routing flow (e.g., due to where middleware is attached relative to request decoding), or that the generated server wiring overwrote middleware list.
   - Consequence: metrics are recorded and traces/logs have request IDs and trace IDs, but they cannot be grouped effectively by operation (e.g., ListPets, CreatePet). The `route` label is present and correct (chi route template), so grouping by route is still possible.

2. OTel collector config warnings/deprecation:
   - The collector initially rejected a `logging` exporter name in the config; replaced with `debug` exporter to make the config compatible with the OTel Collector image version used.
   - The `otlp` exporter alias gets a deprecation warning in collector logs advising `otlp_grpc`, but the collector still worked with `otlp` alias in this run.

3. Docker Compose CLI instability:
   - During heavy build activity, I observed one transient Docker Compose panic (`concurrent map read and map write`) emitted by the CLI progress code. Containers still started; a retry / forced recreate completed successfully.
   - Noted as environment flakiness rather than code issue; I proceeded and ensured containers ended in a healthy state.

4. The generated stubs still return blank echo body values in current sample stubs (`{"echo":""}`). That is part of existing stub behavior and unrelated to observability, but worth noting for test expectations.

## Recommendations / Next steps

1. Fix `operation` propagation so metrics/traces/logs show the OpenAPI operation name (e.g., `Echo`, `EchoPath`, `EchoHeaders`). Potential approaches:
   - Ensure strict middleware (that sets operation in context) is attached and executed for every strict handler path before request handling and before metrics recording.
   - Alternatively, build a route->operation map at server startup and use it in the HTTP middleware to set operation early (this covers request parse errors and pre-middleware failures).
   - Confirm the generated `NewStrictHandlerWithOptions` call receives the operation middleware slice — if `cmd/upd-stubs` or wire code overwrote/nullified middleware lists, update generation/wire accordingly.

2. Harden the OTel collector config and pin explicit versions to avoid deprecation noise (or adapt to the image's recommended exporter names). Optionally use `otlp_grpc` for exporter to Tempo if that’s the current recommended channel.

3. Consider adding an integration test that asserts operation injection: a request to a path results in `operation` being set in metrics and traces.

4. Optionally expand dashboards in Grafana to include:
   - HTTP request count / latency by operation and route
   - Trace search by `request_id`
   - Loki queries by `request_id` and `trace_id`
