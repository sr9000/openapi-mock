# upd-stubs: Генератор OpenAPI-заглушек

`upd-stubs` — CLI-утилита для генерации и безопасного обновления mock-обработчиков OpenAPI на Go.
Инструмент работает в best-effort режиме: обрабатывает все найденные спецификации и не останавливает весь процесс из-за
одной ошибки.

## Ключевые возможности

### 1. Обнаружение спецификаций из `api/`

Поддерживаются оба сценария одновременно:

- `api/<api_name>/openapi.yaml`
- `api/<api_name>/<version>/openapi.yaml` (например, `api/petstore/v3/openapi.yaml`)
  Для каждого найденного spec-path создаются отдельные модули:
- `internal/generated/<rel_path>/...`
- `internal/stubs/<rel_path>/...`
  Это позволяет хранить базовую и версионированную API рядом, не смешивая их код.

### 2. Безопасное обновление существующих stub-файлов

- Добавляются только отсутствующие методы.
- Существующая пользовательская логика в методах не перезаписывается.
- Поиск уже реализованных методов выполняется через Go AST (а не через строковый поиск), поэтому корректно
  поддерживаются:
    - многострочные сигнатуры;
    - переименованные receiver-переменные.
- Для операций с несколькими тегами используется стратегия primary-tag (берется только первый тег), чтобы избежать
  дублирования одного и того же метода в нескольких файлах.

### 3. Надежная генерация сигнатур

- Порядок операций стабилизируется по `StrictServerInterface` из `server.gen.go`.
- Имена методов сверяются с strict-интерфейсом, чтобы избежать рассинхронизации с нормализацией `operationId`.
- Идентификаторы (теги/пакеты) санитизируются до валидных Go-имен.
- Импорты приводятся в порядок через `goimports` для новых и обновленных файлов.

### 4. Управление устаревшими методами

Режим `--prune` не удаляет пользовательский код, а добавляет в файл блок с orphaned-методами, которые больше не
существуют в актуальной OpenAPI-спецификации.

### 5. Генерация Wire-конфигурации

- Генерируется `internal/app/openapi_wire.go` (или путь из `--wire-out`).
- Для импортов используются устойчивые уникальные alias-имена, чтобы избежать коллизий между API/версиями.

## Параметры CLI

```bash
go run ./cmd/upd-stubs \
  --generated-dir internal/generated \
  --stubs-dir internal/stubs \
  --wire-out internal/app/openapi_wire.go \
  --dry-run \
  --prune \
  --verbose
```

Поддерживаемые флаги:

| Флаг              | Описание                                                |
|:------------------|:--------------------------------------------------------|
| `--generated-dir` | Директория с `oapi-codegen` файлами                     |
| `--stubs-dir`     | Куда писать заглушки                                    |
| `--wire-out`      | Путь для генерируемого wire-файла                       |
| `--dry-run`       | Сгенерировать изменения без записи файлов               |
| `--prune`         | Добавлять блок с orphaned-методами при рассинхронизации |
| `--verbose`       | Печатать расширенные диагностические сообщения          |

## Ожидаемая структура проекта

```text
./api/                       # Входные OpenAPI спецификации
└── <api-name>/
    ├── openapi.yaml         # Базовая версия API
    └── v3/
        └── openapi.yaml     # Версионированная API
./internal/generated/        # Код, сгенерированный oapi-codegen
└── <api-name>/...
./internal/stubs/            # Реализации stub-обработчиков
└── <api-name>/...
```

## Проверка edge-cases в тестах

В `cmd/upd-stubs/upd_stubs_test.go` покрыты сценарии:

- AST-поиск методов при многострочной сигнатуре.
- AST-поиск при переименованном receiver.
- Регрессия на отсутствие дублирования метода при update существующего файла.
- Приоритет выбора response-кода.
- Корректная генерация wire после сортировки входных спецификаций.
- Поведение prune-режима.
