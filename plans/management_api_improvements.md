# Management API improvements plan

Date: 2026-05-24

## Goal

Improve management capabilities of `openapi-mock` so integration tests can inspect and control mock execution without restarting Docker containers.

The implementation must be delivered as separate git commits. **Each commit is created only after its Definition of Done passes.** If a DoD item fails, fix the commit contents first; do not commit partial work.

## Current state summary

- Management server lives in `pkg/mgmt/server.go` and currently exposes:
  - `GET /logs`
  - `POST /clear`
  - `DELETE /clear`
  - `GET /doc`
  - `GET /openapi.json`
- Recorded calls are stored in `pkg/recorder.Recorder` as in-memory `[]recorder.CallRecord` with `RequestID`.
- Request IDs are resolved in `pkg/middleware/recording.go` and stored in context through `pkg/observability`.
- Mock OpenAPI specs are generated under `internal/generated/<api>/spec.gen.go`; each generated package exposes `GetSwagger()`.
- Mock app wiring is generated by `cmd/upd-stubs` into `internal/app/openapi_wire.go` and `internal/app/wire_gen.go`.

## Target management API

### Logs

| Method | Path | Behavior |
|:--|:--|:--|
| `GET` | `/logs` | Return all recorded calls. |
| `GET` | `/logs/{request_id}` | Return only records matching `request_id`. |
| `DELETE` | `/logs` | Clear all recorded calls. Replaces old `DELETE /clear`. |

Removed endpoints:

- `POST /clear`
- `DELETE /clear`

### Management docs

Keep existing management documentation endpoints unless a later product decision says otherwise:

| Method | Path | Behavior |
|:--|:--|:--|
| `GET` | `/doc` | Swagger UI for the management API. |
| `GET` | `/openapi.json` | Management OpenAPI document. |

### Mock API docs

Add Swagger UI and JSON OpenAPI access for mocked APIs:

| Method | Path | Behavior |
|:--|:--|:--|
| `GET` | `/docs` | List discovered mock API docs and versions. |
| `GET` | `/docs/{api_name}` | Swagger UI for an unversioned API, the only version, or an index page if multiple versions exist. |
| `GET` | `/docs/{api_name}/openapi.json` | JSON OpenAPI document for unversioned or uniquely resolvable API. |
| `GET` | `/docs/{api_name}/{api_ver}` | Swagger UI for a specific API version. |
| `GET` | `/docs/{api_name}/{api_ver}/openapi.json` | JSON OpenAPI document for a specific API version. |

Version resolution rules:

1. `api/<api_name>/openapi.yaml` registers as `api_name=<api_name>`, `api_ver=""`.
2. `api/<api_name>/<api_ver>/openapi.yaml` registers as `api_name=<api_name>`, `api_ver=<api_ver>`.
3. `GET /docs/{api_name}` serves the unversioned spec when present.
4. If no unversioned spec exists and exactly one version exists, `GET /docs/{api_name}` serves that version.
5. If multiple versions exist and no unversioned spec exists, `GET /docs/{api_name}` returns an HTML/JSON index with links to versions instead of guessing.

### Soft reset

| Method | Path | Behavior |
|:--|:--|:--|
| `POST` | `/reset` | Soft-reset the mock HTTP service without stopping the management or metrics servers and without shutting down Docker. |

Soft reset semantics:

- Gracefully stop the mock HTTP server with a bounded timeout.
- Rebuild the mock router and generated stub handler instances.
- Reattach the same middleware stack.
- Restart the mock HTTP server on the same host/port.
- Clear in-memory recorder logs.
- Clear request-scoped context values.
- Leave the management server, metrics server, process, and container alive.
- Do not reset Prometheus counters unless a separate metrics-reset feature is explicitly added later.

### Request-scoped context values

Add a request-id keyed context-value store controlled by the management API.

| Method | Path | Body | Behavior |
|:--|:--|:--|:--|
| `GET` | `/context-values` | none | Return all configured request-id values. |
| `GET` | `/context-values/{request_id}` | none | Return values for `request_id`, or `{}` when absent. |
| `PUT` | `/context-values` | `{ "req-id": { "name": value } }` | Replace the whole store. |
| `PUT` | `/context-values/{request_id}` | `{ "name": value }` | Replace all values for one request id. |
| `PATCH` | `/context-values` | `{ "req-id": { "name": value } }` | Merge request-id values; add new keys and replace existing keys. |
| `PATCH` | `/context-values/{request_id}` | `{ "name": value }` | Merge values for one request id. |
| `DELETE` | `/context-values` | none | Clear the whole store. |
| `DELETE` | `/context-values/{request_id}` | none or `{ "keys": ["name"] }` | Delete all values for the request id, or only listed keys when body is provided. |

Runtime behavior:

1. Integration test chooses a request id, for example `case-low`.
2. Test calls management API:

   ```bash
   curl -X PUT http://localhost:9000/context-values/case-low \
     -H 'Content-Type: application/json' \
     -d '{"low":10,"high":20}'
   ```

3. Test sends a mock request using the same request id:

   ```bash
   curl http://localhost:8080/example/5 -H 'X-Request-ID: case-low'
   ```

4. The mock handler can read values from `context.Context`:

   ```go
   low := mm.FromCtx(ctx, "low").(int)
   high := mm.FromCtx(ctx, "high").(int)

   if val < low {
       // response 400: not enough
   }
   if val > high {
       // response 400: too much
   }
   // response 200 OK
   ```

Important JSON typing rule:

- Decode JSON with `UseNumber` and normalize recursively.
- Whole JSON numbers fitting Go `int` become `int`, so the sample `. (int)` assertion works for values like `10`.
- Decimal JSON numbers become `float64`.
- Strings, booleans, arrays, objects, and `null` remain available as Go values.

## Proposed internal design

### Management router

- Replace manual method checks in `pkg/mgmt.Server` with explicit route registration.
- Use existing `github.com/go-chi/chi/v5` dependency for path parameters and method-specific routes, or use Go `http.ServeMux` method patterns if keeping dependencies minimal is preferred.
- Keep embedded Swagger UI assets in `pkg/mgmt`.

### Recorder filtering

- Add `Recorder.GetRecordsByRequestID(requestID string) []CallRecord` or filter in `pkg/mgmt`.
- Prefer adding the method to `pkg/recorder` because request-id filtering is core recorder behavior and is independently testable.

### Mock docs registry

Add a registry type owned by `pkg/mgmt`, for example:

```go
type MockDoc struct {
    APIName string
    APIVersion string
    Title string
    SpecJSON func() ([]byte, error)
}

type MockDocsRegistry interface {
    List() []MockDoc
    Find(apiName, apiVersion string) (MockDoc, bool)
    Resolve(apiName string) ResolveResult
}
```

Generate registry construction from `cmd/upd-stubs` because only generated app code knows every `internal/generated/<api>` package at compile time.

Suggested generated function in `internal/app`:

```go
func MockDocs() []mgmt.MockDoc
```

Each entry should call the generated package's `GetSwagger()`, marshal the resolved `*openapi3.T` to JSON, and cache the result after first successful generation.

This avoids runtime filesystem reads and works when the binary is copied into Docker without the source `api/` tree.

### Request context-value package

Add a small package that mock handlers can import with alias `mm`, for example `openapi-mock/pkg/mm`.

Responsibilities:

- Store immutable per-request values in `context.Context`.
- Expose `FromCtx(ctx, name string) any`.
- Expose safe helpers like `Lookup(ctx, name string) (any, bool)` and optionally typed helpers later.
- Provide a concurrency-safe `Store` keyed by request id.
- Provide middleware that reads `observability.RequestID(ctx)` and injects a copy of configured values into context before generated handlers run.

Middleware order requirement:

1. `middleware.Recording` resolves and stores request id in context.
2. Context-value middleware reads that request id and injects values.
3. Generated mock handlers execute.

### Runtime reset controller

Refactor `cmd/openapi-mock/main.go` so the mock HTTP server is managed by a small runtime/controller object instead of being started once inline.

Controller responsibilities:

- Build a new `app.HTTPApp`.
- Start the mock `http.Server`.
- Gracefully stop the current mock `http.Server`.
- Serialize reset calls with a mutex.
- Expose `Reset(ctx context.Context) error` to `pkg/mgmt.Server` through a callback/interface.
- Keep management and metrics servers outside the reset lifecycle.

`pkg/mgmt.New` should receive options rather than only recorder and port, for example:

```go
type Options struct {
    Recorder *recorder.Recorder
    Port string
    MockDocs []MockDoc
    ContextValues *mm.Store
    Reset func(context.Context) error
}
```

## Risks and decisions needing attention

### Mock docs versioning

The repo currently has unversioned examples under `api/echo` and `api/petstore`. The plan assumes future versioned specs will live under `api/<api_name>/<api_ver>/openapi.yaml`. If a different layout is needed, implement the registry with a small mapping function so the route contract stays stable.

### `GET /docs/{api_name}` ambiguity

When multiple versions exist, guessing a default can hide test mistakes. The safer behavior is to serve an index/list page unless an unversioned spec exists or only one version exists.

### Context value typing

JSON has no native integer type distinction, but the example requires `mm.FromCtx(ctx, "low").(int)`. The decoder must use `json.Number` and normalize integer numbers into Go `int`. Tests must lock this behavior down.

### Middleware ordering

Context values depend on the resolved request id. If middleware order is wrong, handlers will see no values. Tests should assert that values are present when `X-Request-ID` is supplied and also when the request id is generated and returned in the response header.

### Soft reset race conditions

Reset must not replace the server while requests are actively mutating shared state without synchronization. Use mutexes around reset lifecycle and already thread-safe stores for recorder/context values.

### Breaking endpoint changes

Removing `/clear` is a breaking management API change. README and OpenAPI docs must clearly direct users to `DELETE /logs`.

## Suggested implementation order within commits

1. Tests first for changed route behavior.
2. Minimal implementation to satisfy tests.
3. Update OpenAPI document.
4. Update README.
5. Run DoD commands.
6. Commit only after DoD passes.
