# Observability upgrade plan for `openapi-mock`

Living document for upgrading generated OpenAPI mock servers with production-style observability.

## 1. Goals

Requested outcomes:

1. Full metrics coverage for every handler.
2. Optional tracing, exportable either to file or to a tracing backend.
3. Mandatory request ID, compatible with incoming headers.
4. `zerolog`-based logging, configurable to console or file, always enriched with request ID and trace ID (when present).
5. A Docker Compose stack that can run the fully featured mock together with metrics, traces, dashboards, and log viewing.

## 2. Current baseline

As of now:

- `pkg/middleware/recording.go` already generates a request ID, records calls, and emits route-based Prometheus metrics.
- `pkg/metrics/metrics.go` exposes HTTP request/error/panic metrics via Prometheus.
- `pkg/middleware/error_handlers.go` records request/response errors.
- `pkg/ctxkeys/structs.go` only carries `RequestID`.
- Stub handlers generated by `cmd/upd-stubs/openapi_stubs.go` use `log.Printf(...)` and only know about request ID.
- `cmd/openapi-mock/main.go` only supports booleans for management, metrics, and plain HTTP logging.
- `docker-compose-grafana.yaml` already runs `openapi-mock` + Prometheus + Grafana, but has no trace backend or log pipeline.

Important limitation in the current implementation:

- Metrics are route-based, but not consistently operation-aware.
- Request IDs are generated server-side, but incoming correlation headers are ignored.
- There is no trace extraction/propagation/export.
- Logs are unstructured and not centrally correlated.
- Error metrics currently include raw `message` labels, which is risky for high-cardinality Prometheus series.

## 3. Recommended target architecture

### 3.1 Request metadata layer

Create a small request metadata layer that is responsible for:

- extracting request ID from allowed incoming headers;
- generating a request ID when missing;
- extracting trace context from incoming headers;
- attaching request metadata to `context.Context`;
- echoing the canonical request ID back in the HTTP response headers;
- exposing helpers for handlers:
  - `RequestID(ctx)`
  - `TraceID(ctx)`
  - `Operation(ctx)`
  - `Logger(ctx)`

Recommended canonical behavior:

- Accept incoming request ID from `X-Request-ID`, `X-Request-Id`, and optionally `X-Correlation-ID`.
- If none is present, generate one.
- Always set `X-Request-ID` in the response.
- Treat request ID as mandatory **inside the system**, not mandatory from the client.

### 3.2 Structured logging

Replace ad-hoc `log.Printf(...)` with a shared `zerolog` setup:

- root logger configured once in `main`;
- sink selectable as `console` or `file`;
- format selectable as `console` or `json`;
- request-scoped logger injected into context with fields:
  - `request_id`
  - `trace_id` (if available)
  - `span_id` (if available)
  - `method`
  - `route`
  - `operation`
  - `status`
  - `duration_ms`

Recommendation:

- access logs should come from middleware, not from every generated handler body;
- handlers should log only business-relevant events via `Logger(ctx)`.

### 3.3 Tracing

Add OpenTelemetry tracing with exporter modes:

- `none`
- `file`
- `otlp-http` (recommended for Docker Compose via collector)

Header compatibility recommendation:

- default: W3C Trace Context (`traceparent`, `tracestate`, `baggage`)
- optional compatibility: B3 single/multi and Jaeger `uber-trace-id`
- configurable via a `TRACE_PROPAGATORS` setting

Recommended behavior:

- if an incoming trace exists, continue it;
- if tracing is enabled and no trace exists, create a root span;
- if tracing is disabled but a trace header exists, still extract `trace_id` for logs;
- add `request_id`, `route`, `operation`, and `http.status_code` as span attributes.

### 3.4 Metrics

Keep Prometheus for metrics, but make labels operation-aware and lower-cardinality.

Recommended metrics:

- `http_server_requests_total{api,operation,method,route,status}`
- `http_server_request_duration_seconds{api,operation,method,route,status}`
- `http_server_inflight_requests{api,operation,method,route}`
- `http_server_errors_total{api,operation,method,route,status,kind}`
- `http_server_panics_total{api,operation,method,route,status}`
- optional later:
  - request size histogram
  - response size histogram

Important change:

- remove raw `message` from Prometheus labels;
- keep messages in logs and traces instead.

### 3.5 Operation awareness

Use two complementary mechanisms:

1. **HTTP middleware** for request ID, logging, recorder integration, route labels, and response propagation.
2. **Strict handler middleware** from generated `server.gen.go` to capture OpenAPI operation names such as `ListPets`, `CreatePet`, etc.

Why both are needed:

- route middleware sees every request;
- strict middleware knows the OpenAPI operation name;
- request parsing errors happen before strict middleware executes, so request error handlers still need route-to-operation metadata.

Recommended solution for request-error coverage:

- generate or build a route lookup map (`method + route -> operation`) and attach it before body decoding;
- fallback to route-only metrics if operation resolution is unavailable.

## 4. Main workstreams

### Workstream A — Request ID and context model

Scope:

- extend `pkg/ctxkeys/structs.go` or introduce a dedicated package such as `pkg/requestmeta` / `pkg/observability`;
- add typed getters/setters for request ID, trace ID, operation, and logger;
- add tests for header precedence and generated fallback.

Impacted files:

- `pkg/ctxkeys/structs.go`
- new `pkg/observability/*.go` (recommended)
- `pkg/middleware/recording.go`
- `pkg/recorder/recorder.go`

Acceptance criteria:

- every recorded request has a request ID;
- every response includes the canonical request ID header;
- handler code can read request ID from context without parsing headers manually.

### Workstream B — Structured logging with `zerolog`

Scope:

- initialize a shared logger in `cmd/openapi-mock/main.go`;
- replace middleware `log.Printf(...)` with structured access logs;
- expose request-scoped logger in context;
- keep sink configurable: stdout or file.

Impacted files:

- `cmd/openapi-mock/main.go`
- `pkg/middleware/recording.go`
- new `pkg/observability/logging.go`
- `cmd/upd-stubs/openapi_stubs.go` (optional generation cleanup)
- existing stubs under `internal/stubs/**` for migration examples

Acceptance criteria:

- all access logs include `request_id`;
- if a trace is present, logs include `trace_id`;
- log format is machine-readable JSON by default in containers.

### Workstream C — Tracing

Scope:

- add OpenTelemetry tracer provider lifecycle;
- support `file` and `otlp-http` exporters;
- configure propagators;
- propagate trace context through request handling.

Impacted files:

- `cmd/openapi-mock/main.go`
- new `pkg/observability/tracing.go`
- `pkg/middleware/recording.go`
- `pkg/middleware/error_handlers.go`
- `internal/app/openapi_wire.go`

Acceptance criteria:

- incoming `traceparent` continues the trace;
- outgoing spans show route and operation attributes;
- tracing can be turned off without breaking logging or request IDs.

### Workstream D — Metrics redesign

Scope:

- refactor metrics labels to use `api`, `operation`, `route`, `status`, and `kind`;
- add in-flight gauge;
- wire metrics into normal, error, and panic paths;
- ensure request parse failures are still counted under the correct route/operation when possible.

Impacted files:

- `pkg/metrics/metrics.go`
- `pkg/middleware/recording.go`
- `pkg/middleware/error_handlers.go`
- `internal/app/openapi_wire.go`

Acceptance criteria:

- every OpenAPI operation increments request counters;
- panics and handler errors are visible without exploding metric cardinality;
- dashboards can group by operation name or route.

### Workstream E — Generator and Wire integration

Scope:

- use strict middleware support from generated handlers to attach operation metadata;
- add operation middleware registration in `internal/app/openapi_wire.go`;
- decide how generated stub methods should log.

Recommendation:

- phase 1: keep existing stub signatures so runtime changes remain backward-compatible;
- phase 2: stop generating inline `log.Printf(...)` blocks in new stub methods and switch documentation/examples to `Logger(ctx)`.

Impacted files:

- `internal/app/openapi_wire.go`
- `internal/app/wire_gen.go`
- `cmd/upd-stubs/openapi_stubs.go`
- `UPD_STUBS.md`
- `README.md`

Acceptance criteria:

- operation name is available in context during handler execution;
- new stub methods no longer depend on stdlib logging;
- existing user code is preserved.

### Workstream F — Docker Compose observability stack

Recommended services:

- `openapi-mock`
- `prometheus`
- `grafana`
- `otel-collector`
- `tempo`
- `loki`
- `promtail` (or `vector`)

Recommended default data flow:

- app metrics: Prometheus scrapes `:9100/metrics`
- app traces: app -> OTLP/HTTP -> OTel Collector -> Tempo
- app logs: app -> stdout JSON -> Promtail -> Loki
- dashboards and explore: Grafana -> Prometheus / Tempo / Loki

Optional profile:

- `dozzle` only if a simple raw Docker log tail UI is desired in addition to Grafana Explore

Impacted files:

- `docker-compose-grafana.yaml` (or replace with `docker-compose-observability.yaml`)
- `prometheus.yaml`
- new collector config under `otel/collector.yaml`
- new `loki/` and `promtail/` configs
- `grafana/provisioning/datasources/datasource.yml`
- `grafana/provisioning/dashboards/*.json`
- `README.md`

Acceptance criteria:

- one compose command brings up metrics, traces, and logs;
- Grafana is pre-provisioned with Prometheus, Tempo, and Loki datasources;
- the mock container can run with console logs or file logs by env config.

## 5. Configuration model

Recommended new env/flag surface:

- `REQUEST_ID_HEADERS` — comma-separated accepted incoming headers
- `REQUEST_ID_RESPONSE_HEADER` — canonical response header, default `X-Request-ID`
- `LOG_ENABLED` — default `true`
- `LOG_FORMAT` — `json|console`
- `LOG_OUTPUT` — `stdout|file`
- `LOG_FILE` — file path when `LOG_OUTPUT=file`
- `LOG_LEVEL` — `debug|info|warn|error`
- `TRACE_ENABLED` — default `false`
- `TRACE_EXPORTER` — `none|file|otlp-http`
- `TRACE_ENDPOINT` — collector endpoint when OTLP is used
- `TRACE_FILE` — path when file exporter is used
- `TRACE_PROPAGATORS` — e.g. `tracecontext,baggage,b3,jaeger`
- `TRACE_SAMPLING_RATIO` — e.g. `1.0`, `0.1`

Keep existing flags/envs for:

- `HOST`
- `PORT`
- `MGMT_PORT`
- `METRICS_PORT`
- `MGMT_ENABLED`
- `METRICS_ENABLED`

## 6. Delivery order

### Phase 1 — Foundation

- request metadata package
- request ID extraction/generation/response propagation
- `zerolog` setup
- structured access log middleware

### Phase 2 — Metrics hardening

- operation-aware labels
- in-flight gauge
- cardinality cleanup for error/panic metrics
- request-error coverage

### Phase 3 — Tracing

- OTel provider + propagators
- file exporter + OTLP exporter
- trace fields in logs and recorder

### Phase 4 — Generator integration

- strict middleware wiring
- generated stub logging cleanup
- docs update for handler-side logging helpers

### Phase 5 — Compose stack

- collector, tempo, loki, promtail
- Grafana datasource provisioning
- ready-made dashboards and explore setup

## 7. Test plan

### Unit tests

- request ID header precedence and generation
- trace header extraction for W3C and optional B3/Jaeger
- logger context contains request ID and trace ID
- metrics label values for success/error/panic
- route-to-operation mapping for request parsing errors

### Integration tests

- `go test ./...` remains green
- HTTP smoke test verifies response has `X-Request-ID`
- trace-enabled test verifies spans are exported to file
- log-enabled test verifies JSON logs include request and trace identifiers
- compose smoke test verifies Grafana datasources and collector reachability

## 8. Risks and migration notes

1. **Prometheus cardinality**
   - current error message labels should not be kept in the new design.

2. **Backward compatibility of stub constructors**
   - do not break existing generated stubs in phase 1.
   - deprecate generated inline logging gradually.

3. **Request parsing errors before strict middleware**
   - this is the main edge case for “every handler” metrics.
   - solve it explicitly instead of assuming strict middleware sees all failures.

4. **Log duplication**
   - avoid both access-log middleware and generated handler entry logs unless there is a strong reason.

5. **Trace disabled mode**
   - logs should still include incoming trace ID when present, even if spans are not exported.

## 9. Recommended first implementation slice

If you want the fastest useful first slice, do this in order:

1. Add request metadata + canonical `X-Request-ID` response header.
2. Switch middleware logging to `zerolog` with request ID fields.
3. Add operation-aware metrics labels and remove raw error-message labels.
4. Add trace context extraction + optional OTLP/file exporter.
5. Expand Docker Compose to Prometheus + Grafana + Tempo + Loki + Collector.

That sequence gives visible value early without forcing a large generator refactor up front.
